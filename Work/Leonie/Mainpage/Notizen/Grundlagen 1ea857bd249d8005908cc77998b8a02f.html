Was ist eine relationale Datenbank?

Eine relationale Datenbank (RDB) ist ein Datenbanksystem, das Daten in Tabellen organisiert, die miteinander in Beziehung stehen. Jede Tabelle besteht aus Zeilen (Datensätzen) und Spalten (Felder). Die Beziehungen zwischen den Tabellen werden durch Primärschlüssel und Fremdschlüssel definiert.

Merkmale:

  • Daten werden in einer strukturierten Form gespeichert (Tabellen).
  • Es wird die Structured Query Language (SQL) verwendet, um Daten zu verwalten und abzufragen.
  • Beziehungen zwischen Tabellen werden durch Schlüssel (z. B. Primär- und Fremdschlüssel) definiert.
  • ACID-Eigenschaften (Atomicity, Consistency, Isolation, Durability) sorgen für Datenintegrität.

Konkrete Hersteller relationaler Datenbanken:

  1. MySQL (Oracle) – Open-Source, weit verbreitet.
  1. PostgreSQL – Open-Source, unterstützt erweiterte Funktionen.
  1. Microsoft SQL Server – Kommerziell, von Microsoft.
  1. Oracle Database – Kommerziell, leistungsstark und weit verbreitet.
  1. SQLite – Leichtgewichtige, dateibasierte Datenbank.
  1. MariaDB – Open-Source, ein Fork von MySQL.

Was ist eine non-relationale Datenbank?

Eine non-relationale Datenbank (auch NoSQL-Datenbank) ist ein Datenbanksystem, das Daten in einer flexibleren Struktur speichert, z. B. als Dokumente, Schlüssel-Wert-Paare, Graphen oder Spaltenfamilien. Sie ist nicht auf Tabellen und Beziehungen angewiesen.

Merkmale:

  • Daten werden in unstrukturierten oder semi-strukturierten Formaten gespeichert.
  • Keine feste Schema-Definition erforderlich (Schema-frei).
  • Skalierbarkeit ist oft horizontal (mehrere Server).
  • Geeignet für große Datenmengen und schnelle Abfragen.
  • Unterstützt verschiedene Datenmodelle (z. B. Dokumente, Graphen, Schlüssel-Wert).

Konkrete Beispiele non-relationaler Datenbanken:

  1. MongoDB – Dokumentenorientiert, speichert Daten im JSON-ähnlichen Format.
  1. Cassandra (Apache) – Spaltenorientiert, hoch skalierbar.
  1. Redis – Schlüssel-Wert-Datenbank, extrem schnell.
  1. Neo4j – Graphdatenbank, ideal für vernetzte Daten.
  1. Couchbase – Dokumentenorientiert, unterstützt auch Schlüssel-Wert.
  1. Amazon DynamoDB – Schlüssel-Wert-Datenbank, von AWS.
  1. Firebase Realtime Database – Echtzeit-Datenbank, von Google.

Wann verwendet man welche?

  • Relationale Datenbanken:
    • Wenn die Daten stark strukturiert sind und Beziehungen zwischen den Daten wichtig sind (z. B. Buchhaltung, CRM-Systeme).
    • Wenn ACID-Eigenschaften erforderlich sind.
  • Non-Relationale Datenbanken:
    • Wenn die Daten unstrukturiert oder semi-strukturiert sind (z. B. JSON-Daten).
    • Wenn hohe Skalierbarkeit und Geschwindigkeit erforderlich sind (z. B. Echtzeit-Anwendungen, Big Data).

Hier ist eine Erklärung der wichtigsten MySQL-Konzepte und Schlüsselwörter, basierend auf deiner Referenz:

Grundlegende MySQL-Befehle

  1. USE
    • Wählt eine Datenbank aus, mit der gearbeitet werden soll.
    USE mondial;
  1. SELECT
    • Dient zum Abrufen von Daten aus einer Tabelle.
    SELECT 'Hello World!';
  1. Alias (AS)
    • Gibt einer Spalte oder einem Ausdruck einen Aliasnamen.
    SELECT Name AS Land, Capital AS Hauptstadt FROM country;

Abfragen von Daten

  1. SELECT * FROM table
    • Ruft alle Spalten einer Tabelle ab.
    SELECT * FROM country;
  1. LIMIT
    • Begrenzung der Anzahl der zurückgegebenen Zeilen.
    SELECT * FROM country LIMIT 10;
SELECT * FROM country LIMIT 20, 10; -- Überspringt 20 Zeilen, gibt die nächsten 10 zurück.
  1. Spaltenauswahl
    • Wählt spezifische Spalten aus.
    SELECT Name, Capital FROM country;
  1. Alias für Spaltennamen
    • Gibt Spalten benutzerdefinierte Namen.
    SELECT Name AS Land, Capital AS Hauptstadt FROM country;

Filterung von Daten

  1. WHERE
    • Filtert Zeilen basierend auf Bedingungen.
    SELECT * FROM country WHERE population > 100000000;
SELECT * FROM country WHERE Name = 'Brazil';
  1. Vergleichsoperatoren
    • =: Gleich
    • <> oder !=: Ungleich
    • <, <=, >, >=: Kleiner, kleiner gleich, größer, größer gleich
    SELECT * FROM country WHERE Name <> 'Brazil';
  1. Logische Operatoren
    • AND: Beide Bedingungen müssen wahr sein.
    • OR: Eine der Bedingungen muss wahr sein.
    • NOT: Negiert eine Bedingung.
    SELECT * FROM country WHERE Name <= 'Brazil' AND Population > 100000000;
SELECT * FROM country WHERE Name <= 'Brazil' OR Population > 100000000;
  1. BETWEEN
    • Überprüft, ob ein Wert in einem Bereich liegt.
    SELECT * FROM country WHERE Population BETWEEN 10000 AND 20000;
  1. LIKE
    • Sucht nach Mustern in Texten.
    • %: Beliebige Anzahl von Zeichen.
    • _: Genau ein Zeichen.
    SELECT * FROM country WHERE Name LIKE 'Uni%'; -- Beginnt mit "Uni"
SELECT * FROM country WHERE Name LIKE '%m'; -- Endet mit "m"
SELECT * FROM country WHERE Name LIKE '%ca%'; -- Enthält "ca"
  1. Längenprüfung
    • Überprüft die Länge eines Strings.
    SELECT * FROM country WHERE LENGTH(Name) < 5;

String-Manipulation

  1. SUBSTR
    • Extrahiert einen Teilstring.
    SELECT *, SUBSTR(Name, 3, 1) FROM country;
SELECT * FROM country WHERE SUBSTR(Name, 3, 1) = 's';

Aggregatfunktionen

  1. COUNT
    • Zählt die Anzahl der Zeilen.
    SELECT COUNT(*) AS anzahlLaender FROM country;

Zusammenfassung

  • USE: Datenbank auswählen.
  • SELECT: Daten abrufen.
  • WHERE: Daten filtern.
  • LIMIT: Ergebnisse begrenzen.
  • LIKE: Muster suchen.
  • BETWEEN: Bereichsprüfung.
  • COUNT: Zeilen zählen.
  • SUBSTR: Teilstrings extrahieren.
  • Alias (AS): Spalten umbenennen.

Wie kann ich in mysql im table city das name dornbirn mit province ausgeben

Um in MySQL aus der Tabelle city den Namen Dornbirn zusammen mit der Spalte province auszugeben, kannst du eine SELECT-Abfrage mit einer WHERE-Bedingung verwenden. Hier ist die entsprechende SQL-Abfrage:

SELECT name, province
FROM city
WHERE name = 'Dornbirn';

Erklärung:

  1. SELECT name, province: Wählt die Spalten name und province aus der Tabelle city aus.
  1. FROM city: Gibt an, dass die Daten aus der Tabelle city abgerufen werden.
  1. WHERE name = 'Dornbirn': Filtert die Zeilen, sodass nur die Zeile mit dem Namen Dornbirn zurückgegeben wird.

Hier sind die Erklärungen zu den SQL-Statements und Kriterien:

1. ORDER BY Statement

Das ORDER BY-Statement wird verwendet, um die Ergebnisse einer Abfrage in aufsteigender (ASC) oder absteigender (DESC) Reihenfolge zu sortieren.

SELECT * FROM users
ORDER BY last_name ASC; -- Sortiert nach Nachnamen in aufsteigender Reihenfolge

2. GROUP BY Statement

Das GROUP BY-Statement gruppiert Zeilen mit denselben Werten in einer oder mehreren Spalten. Es wird oft mit Aggregatfunktionen wie COUNT, SUM, AVG, MAX, MIN verwendet.

SELECT department, COUNT(*) AS employee_count
FROM employees
GROUP BY department; -- Gruppiert nach Abteilungen und zählt die Mitarbeiter

3. JOIN Statement

Das JOIN-Statement kombiniert Zeilen aus zwei oder mehr Tabellen basierend auf einer gemeinsamen Spalte.

SELECT orders.id, customers.name
FROM orders
JOIN customers ON orders.customer_id = customers.id; -- Verknüpft Bestellungen mit Kunden

4. LEFT JOIN Statement

Das LEFT JOIN-Statement gibt alle Zeilen aus der linken Tabelle zurück, auch wenn keine Übereinstimmung in der rechten Tabelle gefunden wird. Nicht übereinstimmende Werte in der rechten Tabelle werden als NULL angezeigt.

SELECT employees.name, departments.name AS department
FROM employees
LEFT JOIN departments ON employees.department_id = departments.id; -- Zeigt alle Mitarbeiter, auch ohne Abteilung

5. IS NULL oder IS NOT NULL Kriterien

Diese Kriterien werden verwendet, um nach NULL-Werten (fehlenden Daten) zu filtern.

SELECT * FROM orders
WHERE delivery_date IS NULL; -- Findet Bestellungen ohne Lieferdatum

SELECT * FROM orders
WHERE delivery_date IS NOT NULL; -- Findet Bestellungen mit Lieferdatum

6. DISTINCT Statement

Das DISTINCT-Statement wird verwendet, um doppelte Werte in einer Spalte zu entfernen und nur eindeutige Werte zurückzugeben.

SELECT DISTINCT country
FROM customers; -- Gibt eine Liste eindeutiger Länder zurück

In MySQL gibt es eine Vielzahl von Funktionen, die in verschiedene Kategorien unterteilt sind. Hier ist eine Übersicht der wichtigsten Funktionen:

1. String-Funktionen

Diese Funktionen arbeiten mit Zeichenketten (Strings).

  • LENGTH(str) – Gibt die Länge eines Strings zurück.
  • LOWER(str) – Wandelt einen String in Kleinbuchstaben um.
  • UPPER(str) – Wandelt einen String in Großbuchstaben um.
  • CONCAT(str1, str2, ...) – Verbindet mehrere Strings.
  • SUBSTR(str, pos, len) oder SUBSTRING(str, pos, len) – Extrahiert einen Teilstring.
  • TRIM(str) – Entfernt Leerzeichen am Anfang und Ende eines Strings.
  • REPLACE(str, from_str, to_str) – Ersetzt Teile eines Strings.
  • LEFT(str, len) – Gibt die ersten len Zeichen eines Strings zurück.
  • RIGHT(str, len) – Gibt die letzten len Zeichen eines Strings zurück.
  • INSTR(str, substr) – Gibt die Position eines Teilstrings zurück.
  • REVERSE(str) – Kehrt einen String um.

2. Numerische Funktionen

Diese Funktionen arbeiten mit Zahlen.

  • ABS(x) – Gibt den Absolutwert von x zurück.
  • CEIL(x) oder CEILING(x) – Rundet x auf die nächste ganze Zahl auf.
  • FLOOR(x) – Rundet x auf die nächste ganze Zahl ab.
  • ROUND(x, d) – Rundet x auf d Dezimalstellen.
  • MOD(x, y) – Gibt den Rest von x / y zurück.
  • POWER(x, y) oder POW(x, y) – Berechnet x hoch y.
  • SQRT(x) – Gibt die Quadratwurzel von x zurück.
  • RAND() – Gibt eine Zufallszahl zwischen 0 und 1 zurück.
  • TRUNCATE(x, d) – Schneidet x auf d Dezimalstellen ab.

3. Datums- und Zeitfunktionen

Diese Funktionen arbeiten mit Datums- und Zeitwerten.

  • NOW() – Gibt das aktuelle Datum und die aktuelle Uhrzeit zurück.
  • CURDATE() – Gibt das aktuelle Datum zurück.
  • CURTIME() – Gibt die aktuelle Uhrzeit zurück.
  • DATE_ADD(date, INTERVAL expr unit) – Fügt einem Datum einen Zeitraum hinzu.
  • DATE_SUB(date, INTERVAL expr unit) – Subtrahiert einen Zeitraum von einem Datum.
  • DATEDIFF(date1, date2) – Gibt die Differenz zwischen zwei Daten in Tagen zurück.
  • DAY(date) – Gibt den Tag eines Datums zurück.
  • MONTH(date) – Gibt den Monat eines Datums zurück.
  • YEAR(date) – Gibt das Jahr eines Datums zurück.
  • HOUR(time) – Gibt die Stunde einer Zeit zurück.
  • MINUTE(time) – Gibt die Minute einer Zeit zurück.
  • SECOND(time) – Gibt die Sekunde einer Zeit zurück.
  • STR_TO_DATE(str, format) – Konvertiert einen String in ein Datum.
  • DATE_FORMAT(date, format) – Formatiert ein Datum in einen bestimmten String.

4. Aggregatfunktionen

Diese Funktionen werden häufig in Kombination mit GROUP BY verwendet.

  • COUNT(expr) – Zählt die Anzahl der Zeilen.
  • SUM(expr) – Gibt die Summe einer Spalte zurück.
  • AVG(expr) – Gibt den Durchschnittswert einer Spalte zurück.
  • MIN(expr) – Gibt den kleinsten Wert einer Spalte zurück.
  • MAX(expr) – Gibt den größten Wert einer Spalte zurück.

5. Logische Funktionen

Diese Funktionen helfen bei der Bedingungsprüfung.

  • IF(condition, true_value, false_value) – Gibt true_value zurück, wenn die Bedingung wahr ist, sonst false_value.
  • CASE – Führt eine bedingte Logik aus.
    CASE
  WHEN condition1 THEN result1
  WHEN condition2 THEN result2
  ELSE resultN
END

6. JSON-Funktionen

Diese Funktionen arbeiten mit JSON-Daten.

  • JSON_OBJECT(key, value, ...) – Erstellt ein JSON-Objekt.
  • JSON_ARRAY(value, ...) – Erstellt ein JSON-Array.
  • JSON_EXTRACT(json_doc, path) – Extrahiert Daten aus einem JSON-Dokument.
  • JSON_CONTAINS(json_doc, value) – Prüft, ob ein JSON-Dokument einen bestimmten Wert enthält.

7. Systemfunktionen

Diese Funktionen liefern Informationen über die Datenbank oder den Server.

  • DATABASE() – Gibt den Namen der aktuellen Datenbank zurück.
  • USER() – Gibt den aktuellen Benutzer zurück.
  • VERSION() – Gibt die MySQL-Version zurück.

Tabellen miteinander verbinden:

Die Abfrage

SELECT c.name AS city_name, c.province AS city_province
FROM city c
INNER JOIN province p ON c.province = p.name
WHERE p.country LIKE 'A';

1. Was macht diese Abfrage?

Diese Abfrage sucht nach Städten aus der Tabelle city, die in Provinzen liegen, die zu einem Land gehören, dessen Name mit dem Buchstaben A beginnt. Die Ergebnisse zeigen den Namen der Stadt und die zugehörige Provinz.

2. Zerlegung der Abfrage

a) SELECT

SELECT c.name AS city_name, c.province AS city_province
  • SELECT: Gibt an, welche Spalten oder Daten du aus der Datenbank abrufen möchtest.
  • c.name: Das ist die Spalte name aus der Tabelle city. Sie enthält den Namen der Stadt.
  • AS city_name: Gibt der Spalte c.name einen neuen Namen (Alias), nämlich city_name. Das macht die Ausgabe übersichtlicher.
  • c.province: Das ist die Spalte province aus der Tabelle city. Sie zeigt, zu welcher Provinz die Stadt gehört.
  • AS city_province: Gibt der Spalte c.province einen neuen Namen (Alias), nämlich city_province.

Beispielausgabe:

city_namecity_province
DornbirnVorarlberg

b) FROM

FROM city c
  • FROM city: Gibt an, dass die Daten aus der Tabelle city stammen.
  • c: Das ist ein Alias für die Tabelle city. Es ist eine Abkürzung, damit du später in der Abfrage einfach c statt city schreiben kannst.

c) INNER JOIN

INNER JOIN province p ON c.province = p.name
  • INNER JOIN: Verbindet zwei Tabellen miteinander. In diesem Fall werden die Tabellen city und province verbunden.
  • province p: Das ist die Tabelle province, die ebenfalls einen Alias (p) bekommt.
  • ON c.province = p.name: Das ist die Bedingung, wie die beiden Tabellen verbunden werden:
    • Die Spalte province aus der Tabelle city (alias c.province) wird mit der Spalte name aus der Tabelle province (alias p.name) verglichen.
    • Das bedeutet: Eine Stadt wird nur dann in die Ergebnisse aufgenommen, wenn ihre Provinz (c.province) mit einer Provinz in der Tabelle province (p.name) übereinstimmt.

Beispiel:

  • In der Tabelle city steht:
    nameprovince
    DornbirnVorarlberg
    InnsbruckTirol
  • In der Tabelle province steht:
    namecountry
    VorarlbergAustria
    TirolAustria
  • Die Verbindung (JOIN) sorgt dafür, dass die Stadt Dornbirn mit der Provinz Vorarlberg und die Stadt Innsbruck mit der Provinz Tirol verknüpft werden.

d) WHERE

WHERE p.country LIKE 'A';
  • WHERE: Filtert die Ergebnisse basierend auf einer Bedingung.
  • p.country: Das ist die Spalte country aus der Tabelle province.
  • LIKE 'A': Prüft, ob der Wert in der Spalte country mit A beginnt.
    • LIKE: Wird verwendet, um Muster zu suchen.
    • A: Bedeutet, dass das Land genau mit dem Buchstaben A beginnt (z. B. "Austria").

Beispiel:

  • Wenn in der Tabelle province steht:
    namecountry
    VorarlbergAustria
    TirolAustria
    BavariaGermany

    Dann werden nur die Provinzen Vorarlberg und Tirol berücksichtigt, weil deren country mit A beginnt.

3. Zusammengefasst

Die Abfrage macht Folgendes:

  1. Sie verbindet die Tabelle city mit der Tabelle province, indem sie die Spalte province in city mit der Spalte name in province abgleicht.
  1. Sie filtert die Ergebnisse so, dass nur Provinzen berücksichtigt werden, deren Land (country) mit dem Buchstaben A beginnt.
  1. Sie gibt den Namen der Stadt (city_name) und die zugehörige Provinz (city_province) aus.

5. Warum ist das nützlich?

  • Du kannst Daten aus mehreren Tabellen kombinieren und filtern.
  • Es hilft, Beziehungen zwischen Tabellen zu analysieren (z. B. welche Städte in welchen Ländern liegen).
  • Du kannst gezielt nur die Daten abrufen, die du benötigst.

Ist es besser bei komplexen Abfragen von Innen nach Außen zu bauen?

Ja, genau! Bei komplexen SQL-Abfragen, insbesondere wenn Unterabfragen verwendet werden, ist es oft sinnvoll, von innen nach außen zu arbeiten. Das bedeutet, dass du zuerst die innerste Abfrage baust, testest und sicherstellst, dass sie korrekt funktioniert, bevor du sie in eine äußere Abfrage einfügst.

Warum von innen nach außen arbeiten?

  1. Schrittweises Debugging:
    • Du kannst jede Unterabfrage einzeln testen und sicherstellen, dass sie die erwarteten Ergebnisse liefert.
    • Das macht es einfacher, Fehler zu finden und zu beheben.
  1. Bessere Lesbarkeit:
    • Wenn du die inneren Abfragen zuerst baust, kannst du die Logik klarer verstehen und die äußeren Abfragen darauf aufbauen.
  1. Komplexität reduzieren:
    • Indem du die Abfrage schrittweise aufbaust, vermeidest du, dass du dich in der Verschachtelung verlierst.

Wie baust du eine Abfrage von innen nach außen?

Beispiel: Finde das Land mit den meisten Religionen

  1. Innerste Abfrage: Zähle die Religionen pro Land
    • Beginne mit der Abfrage, die die Anzahl der Religionen pro Land berechnet:
    SELECT c2.Name, COUNT(r2.Name) AS religion_count
FROM religion r2
INNER JOIN country c2 ON r2.Country = c2.Code
GROUP BY c2.Name;
    • Ergebnis: Eine Liste aller Länder mit der Anzahl ihrer Religionen:
      Name           | religion_count
-------------------------------
Germany        | 2
France         | 3
United States  | 4
  1. Nächste Ebene: Finde die maximale Anzahl von Religionen
    • Verwende die Ergebnisse der ersten Abfrage, um die maximale Anzahl von Religionen zu berechnen:
    SELECT MAX(religion_count)
FROM (
    SELECT c2.Name, COUNT(r2.Name) AS religion_count
    FROM religion r2
    INNER JOIN country c2 ON r2.Country = c2.Code
    GROUP BY c2.Name
) AS countRel;
    • Ergebnis: Die maximale Anzahl von Religionen:
      MAX(religion_count)
-------------------
4
  1. Äußere Abfrage: Filtere Länder mit der maximalen Anzahl
    • Verwende die maximale Anzahl aus der zweiten Abfrage, um die Länder zu filtern:
    SELECT c.Name, COUNT(r.Name) AS religion_count
FROM religion r
INNER JOIN country c ON r.Country = c.Code
GROUP BY c.Name
HAVING COUNT(r.Name) = (
    SELECT MAX(religion_count)
    FROM (
        SELECT c2.Name, COUNT(r2.Name) AS religion_count
        FROM religion r2
        INNER JOIN country c2 ON r2.Country = c2.Code
        GROUP BY c2.Name
    ) AS countRel
);
    • Ergebnis: Das Land mit der höchsten Anzahl von Religionen:
      Name           | religion_count
-------------------------------
United States  | 4

Vorteile dieser Methode:

  1. Einfaches Testen:
    • Du kannst jede Abfrage einzeln ausführen und sicherstellen, dass sie korrekt funktioniert, bevor du sie in die nächste Ebene einfügst.
  1. Klarheit:
    • Du verstehst genau, was jede Abfrage macht, und kannst die Logik Schritt für Schritt aufbauen.
  1. Fehler vermeiden:
    • Wenn ein Teil der Abfrage nicht funktioniert, kannst du den Fehler isolieren und beheben, ohne die gesamte Abfrage zu überarbeiten.

Zusammenfassung:

Ja, bei komplexen Abfragen mit Unterabfragen ist es eine sehr gute Praxis, von innen nach außen zu arbeiten. Es hilft dir, die Logik zu verstehen, Fehler zu vermeiden und die Abfrage schrittweise aufzubauen.
